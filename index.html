🧩 Core Concepts

พื้นฐานที่ต้องรู้ก่อนเริ่มใช้ GitHub Actions

📋

Workflow

กระบวนการอัตโนมัติที่กำหนดใน YAML file เก็บไว้ที่ .github/workflows/

⚡

Event

สิ่งที่ trigger workflow ให้ทำงาน เช่น push, pull_request, schedule

💼

Job

ชุดของ steps ที่ทำงานบน runner เดียวกัน สามารถรัน parallel ได้

👣

Step

task ย่อยใน job — รัน command หรือเรียกใช้ action

🔌

Action

custom application ที่ reusable ได้ มีทั้งสำเร็จรูปและสร้างเอง

🖥️

Runner

server ที่รัน workflow — ใช้ GitHub-hosted หรือ self-hosted ก็ได้

📝 Workflow Syntax

โครงสร้างไฟล์ Workflow YAML ที่เก็บใน .github/workflows/

yaml 
name: CI Pipeline                    # ชื่อ workflow

on:                                  # Events ที่ trigger
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:                         # กำหนดสิทธิ์
  contents: read

jobs:                                # รายการ jobs
  build:
    runs-on: ubuntu-latest           # runner ที่ใช้

    steps:
      - name: Checkout code
        uses: actions/checkout@v4    # ใช้ action สำเร็จรูป

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci                  # รัน command

      - name: Run tests
        run: npm test

      - name: Build
        run: npm run build

⚡ Events & Triggers

วิธีกำหนดว่า workflow จะถูก trigger เมื่อไหร่

yaml 
on:
  push:
    branches: [main, develop]
    paths:
      - 'src/**'              # trigger เมื่อไฟล์ใน src/ เปลี่ยน
    tags:
      - 'v*'                  # trigger เมื่อ push tag v*

  pull_request:
    types: [opened, synchronize, reopened]
yaml 
on:
  schedule:
    - cron: '0 2 * * 1'      # ทุกวันจันทร์ เวลา 02:00 UTC
    - cron: '30 9 * * *'     # ทุกวัน เวลา 09:30 UTC

# Cron Format: minute hour day month weekday
# ┌─── minute (0-59)
# │ ┌─── hour (0-23)
# │ │ ┌─── day of month (1-31)
# │ │ │ ┌─── month (1-12)
# │ │ │ │ ┌─── day of week (0-6, Sun=0)
# * * * * *
yaml 
on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Target environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production
      debug:
        description: 'Enable debug logging'
        required: false
        type: boolean
Event คำอธิบาย
release เมื่อมี release ใหม่ถูกสร้าง
issues เมื่อมี issue ถูกสร้างหรือแก้ไข
workflow_call เรียกจาก workflow อื่น (reusable workflows)
repository_dispatch trigger จาก API ภายนอก
workflow_run trigger เมื่อ workflow อื่นเสร็จ
create / delete เมื่อ branch หรือ tag ถูกสร้าง/ลบ

🔧 Jobs & Steps

การจัดการ jobs, dependencies, conditions และ outputs

yaml 
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  build:
    needs: test                      # รอ test เสร็จก่อน
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run build

  deploy:
    needs: [test, build]             # รอทั้ง test และ build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - run: echo "Deploying..."
💡
Tip: ใช้ needs เพื่อสร้าง dependency chain ระหว่าง jobs — jobs ที่ไม่มี dependency จะรัน parallel อัตโนมัติ
yaml 
steps:
  - name: Deploy to production
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    run: ./deploy.sh

  - name: Notify on failure
    if: failure()                  # รันเมื่อ step ก่อนหน้า fail
    run: echo "Build failed!"

  - name: Always run cleanup
    if: always()                   # รันทุกกรณี
    run: ./cleanup.sh

  - name: Only on success
    if: success()                  # รันเมื่อสำเร็จ (default)
    run: echo "All good!"
yaml 
jobs:
  prepare:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.get_version.outputs.version }}
    steps:
      - id: get_version
        run: echo "version=1.2.3" >> $GITHUB_OUTPUT

  deploy:
    needs: prepare
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying v${{ needs.prepare.outputs.version }}"

🏪 Actions Marketplace

Actions สำเร็จรูปยอดนิยมที่ใช้บ่อย

Action ใช้ทำอะไร
actions/checkout@v4 Clone repository เข้ามาใน runner
actions/setup-node@v4 ติดตั้ง Node.js version ที่ต้องการ
actions/setup-python@v5 ติดตั้ง Python version ที่ต้องการ
actions/cache@v4 Cache dependencies เพื่อเพิ่มความเร็ว
actions/upload-artifact@v4 Upload build artifacts
actions/deploy-pages@v4 Deploy ไปยัง GitHub Pages
yaml — Custom Composite Action 
# .github/actions/greeting/action.yml
name: 'Greeting Action'
description: 'Greet someone'
inputs:
  who-to-greet:
    description: 'Who to greet'
    required: true
    default: 'World'
runs:
  using: 'composite'
  steps:
    - run: echo "Hello ${{ inputs.who-to-greet }}!"
      shell: bash

🔐 Secrets & Variables

จัดการข้อมูลลับและค่า configuration ใน workflows

🔒 Secrets

ข้อมูลลับที่ถูก encrypt เช่น API keys, passwords — ถูก mask ใน logs อัตโนมัติ

yaml 
env:
  API_KEY: ${{ secrets.API_KEY }}
  DB_PASS: ${{ secrets.DB_PASSWORD }}

📊 Variables

ค่าที่ไม่ลับ เช่น environment name, feature flags — มองเห็นได้ใน logs

yaml 
env:
  APP_ENV: ${{ vars.APP_ENV }}
  API_URL: ${{ vars.API_URL }}
⚠️
สำคัญ: อย่าเขียน secrets ลงใน workflow files โดยตรง! ใช้ Settings > Secrets and variables > Actions เพื่อเพิ่ม secrets ผ่าน GitHub UI เท่านั้น

🌍 Environments

จัดการ deployment environments พร้อมระบบ protection rules

yaml 
jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.example.com
    steps:
      - run: echo "Deploy to staging"

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://example.com
    steps:
      - run: echo "Deploy to production"
  • 🛡️
    Protection Rules ต้องมีคน approve ก่อน deploy — ตั้งค่า required reviewers ได้
  • ⏰
    Wait Timer รอเวลาที่กำหนดก่อน deploy จริง เช่น 15 นาที
  • 🔑
    Environment Secrets secrets เฉพาะ environment — production ใช้คนละค่ากับ staging
  • 🌿
    Branch Policies จำกัดว่า branch ไหนสามารถ deploy ไป environment นี้ได้

🔄 Matrix Strategy

ทดสอบโค้ดบนหลาย OS, versions, และ configurations พร้อมกัน

yaml 
jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
        exclude:                       # ข้ามบาง combination
          - os: macos-latest
            node-version: 18
        include:                       # เพิ่ม combination พิเศษ
          - os: ubuntu-latest
            node-version: 22
            experimental: true
      fail-fast: false              # ไม่หยุดถ้า job อื่น fail

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test
📊
ตัวอย่างข้างต้นจะสร้าง 8 jobs (3 OS × 3 Node versions - 1 excluded) ที่รัน parallel — ทำให้ทดสอบครอบคลุมหลาย environment ได้อย่างรวดเร็ว

📦 Caching & Artifacts

เพิ่มความเร็ว workflow ด้วย caching และส่ง artifacts ระหว่าง jobs

yaml 
steps:
  - uses: actions/cache@v4
    with:
      path: ~/.npm                # path ที่จะ cache
      key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
      restore-keys: |
        ${{ runner.os }}-node-    # fallback keys

  - run: npm ci                   # จะเร็วขึ้นถ้ามี cache
yaml 
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: build-output
          path: dist/
          retention-days: 7          # เก็บไว้ 7 วัน

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: build-output
          path: dist/
      - run: ls -la dist/            # ใช้ artifact จาก build job

✅ Best Practices

แนวทางปฏิบัติที่ดีสำหรับการใช้ GitHub Actions

Security

🔒 ความปลอดภัย

  • ใช้ pinned versions สำหรับ actions (@v4 หรือ commit SHA)
  • กำหนด permissions ให้น้อยที่สุด
  • อย่า hardcode secrets ใน workflow files
  • ใช้ CODEOWNERS เพื่อควบคุมการแก้ไข workflow
Performance

⚡ ประสิทธิภาพ

  • ใช้ caching สำหรับ dependencies
  • ใช้ paths filter เพื่อ trigger เฉพาะที่จำเป็น
  • ใช้ concurrency เพื่อยกเลิก runs ซ้ำ
  • แยก jobs ออกเพื่อรัน parallel ได้
Maintenance

📋 ดูแลรักษา

  • ใช้ reusable workflows สำหรับ logic ซ้ำ
  • ใช้ composite actions สำหรับ steps ที่ใช้บ่อย
  • เขียน comments ใน workflow files
  • ใช้ meaningful names สำหรับ jobs/steps
yaml — Concurrency Control 
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true          # ยกเลิก run เก่าถ้ามี run ใหม่
